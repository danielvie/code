What MCP Is

Model Context Protocol (MCP) is a standard way for an AI application to discover and call external capabilities without hardcoding every integration into the model client.

Problem without MCP MCP approach
Every agent hardcodes filesystem, browser, database, and API integrations. Each integration runs as an MCP server with discoverable tools.
Tool schemas are copied manually into the agent. The client asks the server for tool metadata at startup.
Changing an integration requires changing the agent core. Changing an integration usually means changing only the MCP server or config. 
# Simple idea
Agent host → MCP client → MCP server → external system
Model sees server tools as callable functions.
Rule: MCP is not the model. MCP is the protocol layer that lets a model-powered host discover and call external capabilities safely and consistently.

Mental Model

The most useful way to understand MCP is to separate the process roles from the things they expose.

Term Meaning Example
Host The application using the model and MCP clients. IDE agent, desktop chat app, CLI agent.
Client The protocol object that connects to one MCP server. A Python ClientSession.
Server A process that exposes tools, resources, or prompts. Filesystem server, browser server, database server.
Tool A callable operation with JSON input and output content. random_number, navigate, query_database.
Resource Readable contextual data, often URI-addressed. Documents, repo files, logs.
Prompt A reusable server-provided prompt template. Code review prompt, incident summary prompt. 
{
  "mcpServers": {
    "browser": {
      "command": "npx",
      "args": ["-y", "chrome-devtools-mcp@latest"]
    }
  }
}
Design implication: one host can connect to many servers, but each server should have a narrow responsibility so tool names and permissions stay understandable.

Python MCP Server

A Python MCP server is just a process that registers functions as MCP tools and runs a transport, commonly stdio for local agents.

Minimal Python server

from random import randint
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("random-number")

@mcp.tool()
def random_number() -> int:
    """Return a random integer from 0 to 1000."""
    return randint(0, 1000)

if __name__ == "__main__":
    mcp.run()

Run from config

{
  "mcpServers": {
    "random": {
      "command": "uv",
      "args": ["run", "python", "-m", "mcp_servers.random_mcp_server"]
    }
  }
}
Line Purpose
FastMCP("random-number") Creates the MCP server instance and names it for diagnostics.
@mcp.tool() Publishes a Python function as a model-callable MCP tool.
mcp.run() Starts the server transport so an MCP client can connect.
Rule: tool docstrings matter. They become part of the tool description the model uses to decide when to call the tool.

TypeScript MCP Server

A TypeScript MCP server follows the same shape: define a server, register tool schemas, implement handlers, and run stdio transport.

Minimal TypeScript server

import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  CallToolRequestSchema,
  ListToolsRequestSchema,
} from "@modelcontextprotocol/sdk/types.js";

const server = new Server(
  { name: "random-number", version: "1.0.0" },
  { capabilities: { tools: {} } },
);

server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{
    name: "random_number",
    description: "Return a random integer from 0 to 1000.",
    inputSchema: { type: "object", properties: {} },
  }],
}));

server.setRequestHandler(CallToolRequestSchema, async (request) => {
  if (request.params.name !== "random_number") throw new Error("Unknown tool");
  const value = Math.floor(Math.random() * 1001);
  return { content: [{ type: "text", text: String(value) }] };
});

await server.connect(new StdioServerTransport());

Package command

{
  "type": "module",
  "bin": { "random-mcp": "dist/server.js" },
  "dependencies": {
    "@modelcontextprotocol/sdk": "latest"
  }
}
Python FastMCP TypeScript SDK
@mcp.tool() derives schema from the function. ListToolsRequestSchema returns explicit JSON schemas.
mcp.run() starts the default transport. server.connect(new StdioServerTransport()) starts stdio.
Best for quick local tools. Best when you want explicit protocol control.
Compatibility: TypeScript MCP servers are still external processes. A Python agent can call a TypeScript MCP server as long as both speak MCP over the configured transport.

Agent Client Integration

An MCP-capable agent loads server configs, starts server processes, lists tools, converts them into model tool specs, and routes tool calls back to the right server. 

async with MCPToolbox("mpc.json") as mcp_toolbox:
    tools = local_tools + mcp_toolbox.get_tools()
    response = client.chat.completions.create(
        model="...",
        messages=messages,
        tools=tools,
    )
tool_name = tool_call.function.name
if mcp_toolbox.has_tool(tool_name):
    result = await mcp_toolbox.execute(
        tool_name=tool_name,
        arguments=tool_call.function.arguments,
    )
else:
    result = run_local_tool(tool_call)
Client responsibility Why it exists
Start server process The MCP server is independent from the main agent process.
List server tools The model needs names, descriptions, and JSON schemas.
Track tool ownership The tool call must be routed to the server that owns it.
Stringify results The model consumes tool results as conversation messages.

Inside a beginner-friendly MCP client

A minimal client adapter can be understood as four pieces of state: active server sessions, model-visible tool definitions, a map from tool name to server name, and a cleanup stack.

Name Type Beginner meaning
server_sessions dict[str, ClientSession] One live connection per MCP server.
openai_tools list[ChatCompletionFunctionToolParam] The converted tool schemas given to the model.
tool_server_names dict[str, str] A lookup table that answers: “which server owns this tool?”
exit_stack AsyncExitStack A cleanup helper that closes all MCP sessions when the agent exits.

Startup flow

At startup, the client does not know what tools exist. It must start every configured server and ask each server for its tool list. 

async def load_servers(config_path):
    config = read_json(config_path)

    for server_name, server_config in config["mcpServers"].items():
        session = await start_server(server_config)
        server_sessions[server_name] = session
        await add_server_tools(server_name, session)
Step What happens Why it matters
Read config Load commands from mpc.json. The agent can change servers without changing model-loop code.
Start process Run uv, npx, or another command over stdio. MCP servers stay external to the main program.
Initialize session Perform MCP handshake with ClientSession.initialize(). Both sides agree protocol capabilities.
List tools Call session.list_tools(). The model receives current tool names, descriptions, and schemas.

Tool name mapping

The model needs globally unique tool names. The tutorial client builds names with mcp__server__tool. 

def make_openai_tool_name(server_name, mcp_tool_name):
    return f"mcp__{server_name}__{mcp_tool_name}"

# examples:
# random + random_number → mcp__random__random_number
# chrome-devtools + navigate → mcp__chrome-devtools__navigate
Why prefix names? Two servers may both expose a tool named read. Prefixing with the server name prevents collisions and makes dispatch simple.

Dispatch flow

When the model calls a tool, the client checks whether the name belongs to MCP. If yes, it strips the prefix and calls the original MCP tool on the owning server session. 

async def execute(tool_name, arguments):
    server_name = tool_server_names[tool_name.lower()]
    session = server_sessions[server_name]

    mcp_tool_name = tool_name.split("__", 2)[2]
    mcp_arguments = json.loads(arguments or "{}")

    result = await session.call_tool(mcp_tool_name, mcp_arguments)
    return content_to_text(result.content)

Result conversion

MCP tools return content blocks. The agent needs one text string so it can append the result as a normal tool message. 

def content_to_text(content):
    parts = []
    for item in content:
        if getattr(item, "text", None) is not None:
            parts.append(item.text)
        else:
            parts.append(json.dumps(item.model_dump()))
    return "\n".join(parts)
Boundary: the agent should not import server implementation modules directly. It should talk to them through MCP, the same way it would talk to an external browser or database server.

Testing and Debugging

MCP failures usually come from config, process startup, schema mismatch, or the model choosing the wrong tool.

Prompt-based smoke tests

test:mcp:random:
  desc: "Test the random MCP server"
  cmds:
    - uv run -m app.main -p "Call the random MCP tool. Print the tool result exactly as returned."

Common failures

Symptom Likely cause Fix
No tools appear Config path missing or server failed during initialize. Run the server command manually and inspect stderr.
Wrong tool called Prompt is vague or tool descriptions overlap. Name the target server/tool in the prompt and improve descriptions.
Old code is used Packaged command cache or stale build artifacts. Use uv run python -m module for local development.
Browser does not visibly open Server is configured with --headless. Remove headless mode for visual debugging. 
# Useful checks
uv run python -m compileall app mcp_servers
task test:mcp:random
task test:mcp:chrome-devtools
Testing rule: separate protocol tests from model-choice tests. First verify the MCP tool can run; then verify the prompt makes the model choose it.
